Scheduling-part I

Posted on 2020-05-27 Edited on 2020-05-28 In OS

我们现在需要了解OS scheduler的一系列的调度策略(scheduling policies)。调度的起源，实际上早于计算机系统，一开始是用于操作管理领域。现在我们的问题是，如何建立调度策略？

workload assumption

作者在这里做了一些关于进程（有时也叫jobs)的假设:

  1. Each job runs for the same amount of time.
  2. All jobs arrive at the same time.
  3. Once started, each job runs to completion.
  4. All jobs only use the CPU(i.e., the perform no I/O)
  5. the run-time of each job is known.

上面的假设都是不现实的(unrealistic)，所以你会感到困惑：为什么每个进程的运行时间都是已知的呢？后面我们将逐步放开限制。

Scheduling Metrics

A metrics is just something that we use to measure something, and there are a number of different metrics that make sense in scheduling.

Let us simply our life by simply have a single metric: turnaround time(周转时间). The turnaround time of a job is defined as the time at which the job completes minus the time at which the job arrived in the system. More formally, the turnaround time TturnarroundT_{turnarround}Tturnarround​ is:

Tturnarround=Tcompletion−TarrivalT_{turnarround} = T_{completion} - T_{arrival} Tturnarround​=Tcompletion​−Tarrival​

Another metric of interest is fairness, but we focus on turnaround time in this chapter.

A New Metric: Responce Time

Response time is defined as the time from when the job arrives in a system to the first time it is scheduled. More formally:

Tresponse=Tfirstrun−TarrivalT_{response} = T_{firstrun} - T_{arrival} Tresponse​=Tfirstrun​−Tarrival​

例如，刚才的FTCF的例子中，A的响应时间是0，B的响应时间也是0，C的响应时间是10.

Round Robin

为了解决这个问题，我们介绍一个新的调度算法，被称为Round-Robin (RR)(轮转调度) scheduling.

基本的思想很简单: RR运行一个进程，时间为1个时间片(time slice)，然后切换到下一个在运行队列的进程。 一直重复如此，直到所有的进程都完成。

基于RR的思想，RR有时也被叫做time-slicing.

注意: time slice的大小必须是timer-interupt周期的整数倍。

同时，RR算法在周转时间方面表现得也很糟糕，因为RR算法是尽可能地拉长每个进程。

Incorporating I/O

首先，我们先松弛假设4(All jobs only use the CPU)，每个程序都可以执行I/O操作。

当一个进程提出I/O请求的时候，调度器应该清楚地做出决定，因为当前运行的进程在I/O过程中不会使用CPU.

当I/O完成的时候，调度器也必须做出决定。因为那样的话，一个中断会发生，然后OS将进程转变为ready状态。我们此时应该如何对待每一个进程呢？

为了更好地理解这个问题，我们在这里举个例子。假设有两个进程，A和B,分别都运行50ms的CPU时间。但是，A会每运行10ms后发起I/O操作，每个I/O操作持续10ms，但是B只是简单地使用CPU，没有I/O操作。调度器先运行A，然后运行B。

显然，只允许一个进程，然后再运行另一个而不考虑I/O显然没有任何意义。

一个通常的做法是把每一个A的10ms的子工作(sub-job)看成是一个独立地工作。

因此，一开始的时候，调度器的选择是一个10-ms的A还是一个50ms的B. 在STCF的调度策略下，应该选择A。当第一个A的sub-job运行完成后，此时只有B剩下，然后B开始运行。接着B运行了10ms之后，一个A的新的sub-job被提交到运行队列，根据STCF策略，此时会运行A的sub-job。一直重复执行上述操作。
